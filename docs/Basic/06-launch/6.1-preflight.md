# 6.1 为什么把“上线”放在最后，以及上线前检查

第 1 章做完时，你当然已经可以预览一个版本，但那个阶段更像原型。第 6 章才是正式亮相。之所以把上线放到最后，不是因为它特别神秘，而是因为它本来就应该建立在前面几章的结果之上：页面已经有基本样子，内容已经更完整，数字分身也更像你了。只有到了这个程度，链接才真正值得被别人点开。

## 为什么不是第 1 章做完就立刻发出去

因为第 1 章解决的是“先让它出现”，不是“它已经准备好接受别人访问”。一个原型能预览，不代表它已经值得公开。很多东西都还在早期：内容可能太空，手机端可能还不稳，聊天入口可能还不够清楚，数字分身也可能还不够像你。

到了第 6 章再上线，反而是对整个项目更负责的节奏。你不是在拖延，而是在等它长到一个值得被别人认真看一眼的状态。

## 上线前先做一次很朴素的自查

所以上线前最重要的，不是马上点部署按钮，而是先做一次很朴素的自查。页面能不能打开，手机端是不是正常，聊天是不是能用，个人信息是不是已经补全，环境变量和密钥是不是放在正确位置，这些都该先看一遍。它不需要变成正式测试体系，但至少要保证你不是把一个明显还没准备好的版本直接公开出去。

一个很够用的上线前清单是：

- 页面能正常打开，没有明显报错
- 手机端阅读没有严重错位
- 数字分身可以正常回复
- 你的名字、介绍、作品、联系方式已经不是空白
- API Key 没有写死在代码里，环境变量位置正确

## 上线不是附加任务，而是闭环的一部分

这一步的意义，不只是减少翻车，更是在提醒你：上线不是额外附加任务，而是作品闭环的一部分。它不是“顺手发一下”，而是“这个版本现在已经值得进入真实环境了”。
