# 0.2 谁适合学，以及怎么学最有效

## 先把期待放到正确的位置

一套教程是否适合你，不取决于它“好不好”，而取决于它和你当前阶段是否匹配。

这套基础篇的目标非常明确：

> 帮零基础或非技术背景的读者，完成第一次从想法到上线的完整作品闭环。

如果你的目标和这个很接近，那这套基础篇大概率会对你有用。

## 这套基础篇适合谁

| 你是谁 | 你会从这里得到什么 |
|--------|----------------------|
| 零基础学生 | 第一次知道“我也能把想法做成东西” |
| 产品 / 运营 / 设计背景读者 | 把过去只能写在文档里的想法变成可演示作品 |
| 想做个人主页 / 作品集的人 | 做出一个真的可以分享出去的个人项目 |
| 想低成本试项目的人 | 用最小闭环完成第一次验证，而不是一直停在想象里 |

它尤其适合这样的人：你不是来考技术理论的，你想先做出一个作品，再决定要不要继续深入，也愿意边做边学，而不是等“准备好了再开始”。

## 这套基础篇不太适合谁

如果你当前最强烈的目标是下面这些，那这套基础篇可能不是最优入口：

| 你的目标 | 为什么这不是最佳起点 |
|----------|------------------------|
| 系统学习前端 / 后端 / 全栈原理 | 基础篇不会按传统学科方式系统展开 |
| 一开始就做复杂 SaaS / 多人协作后台 | 基础篇主线更强调第一件作品和最小闭环 |
| 想深入 Git 协作、部署架构、域名、DNS、CI/CD | 这些会留到进阶版系统讲 |

这不是说你“不能学”，而是说：如果你对这些内容的需求已经非常明确，基础篇可能会让你觉得“为什么这里收得这么紧”。

## 学完之后，你会到什么程度

学完基础篇后，你大概会到这样一个状态：你能做出一个真正可以展示的 AI 作品，知道怎么用 AI 去继续修改它，也知道遇到卡点时该如何继续推进；与此同时，你会对 Git、API、部署、安全这些词建立起一种“够用理解”。

但你还不会自动变成专业全栈工程师，也不会一下拥有大型架构设计能力。基础篇更现实的目标，是把你从“完全没有作品”带到“已经有第一个完整作品，而且知道怎么继续往下走”。

## 怎么学最有效

更适合这套内容的打开方式，不是从头到尾连续阅读，而是：

> 一边做，一边读；遇到问题时，再回到对应位置查。

### 三种推荐路径

如果你只是想先试试看，最适合走“体验路径”：看完第 0 章，直接做第 1 章，先拿到一个能预览、能聊天的 `v1` 原型。如果你想完整走一遍从想法到上线的闭环，那就按顺序完成第 0 章到第 6 章。要是你本来就习惯边做边查，也可以走“边做边跳读”的路子：第 0 章看完后先做第 1 章，卡到哪里，再回头查对应章节或附录。

### 如何使用附录

附录不是预习任务，更像工具箱。看到报错了，就去查“常见错误与问 AI 流程”；需要一些界面表达词汇，就去查“UI 速查卡”；忘了 Git 的最小动作，再去翻“Git 最小操作卡”。它不是拿来先通读一遍的，而是在你需要的时候，能马上把你往前推一把。

### 每一章最重要的不是看完，而是交付

更有效的方式是：先看本章要做成什么，再动手做，用验收清单确认结果，然后继续下一章。

如果你只是阅读而不动手，很容易产生一种错觉：“我好像懂了。”但一旦真正开始做，就会发现自己并没有建立起推进能力。

## 给自己留出一个现实的节奏

你不需要一口气做完全部章节。更好的方式，是给自己一个能持续下去的节奏：一次只完成一个小节，或者一次只完成一章里最重要的动作，但要保证每次结束时，作品都比开始时更往前一点。

比“今天看了很多”更重要的，是“今天作品真的往前走了一步”。接下来，我们来聊最现实的问题：如果你中途卡住了，该怎么办。
