# 4.1 为什么先学“存档”，以及 Git 的最小闭环

从这一章开始，你对项目的修改会变得更频繁，也更容易牵一发而动全身。你可能会加一个作品区，顺手改到布局；可能会补联系方式，不小心把首屏间距弄乱；也可能会调整聊天入口，结果别的地方一起变形。到这个阶段，如果还没有任何“后悔药”，你很容易在改动时变得畏手畏脚，或者干脆因为一次改崩就把前面的努力一起拖下去。

所以第 4 章先不急着加内容，而是先学最小的存档闭环。这里的 Git 不需要被理解成什么专业协作体系，它在基础版里只有一个非常朴素的作用：让你在大胆改的时候，有地方可以退回来。你只要先掌握三个动作就够了：做一个存档点，看看自己之前存过哪些版本，以及在必要时回到上一个可用版本。到这个程度，就已经非常够用。

## 为什么现在学，而不是更早或更晚

第 1 章和第 2 章时，项目还很小，改动也相对克制。到了第 4 章，情况开始变了：你要加内容模块，要调整页面结构，要改聊天入口的使用方式，很多修改都不是一句文案那么轻。也正因为改动开始变大，存档才第一次变得非常值回票价。

这里的顺序很重要：不是项目彻底崩了才去学怎么回退，而是在改动真正复杂起来之前，先给自己留后路。

## 基础版里的 Git，只负责一件事

如果你现在一听到 Git 就紧张，也可以把这件事想得更现实一点：它不是要你突然学会分支、PR、合并冲突，而只是给项目装一个最基础的时光机。甚至连这些操作，你都不一定要自己硬记命令。你完全可以让 AI 帮你做，重点是你知道自己现在为什么需要它，以及你希望它替你完成什么。

对基础版来说，Git 只负责三件事：

| 动作 | 你现在把它理解成什么 |
|------|--------------------|
| 存一次 | 给当前稳定版本打一个可回来的点 |
| 看历史 | 看看自己之前存过哪些版本 |
| 退回来 | 某次改崩时，回到上一个可用版本 |

到这个程度，就已经很够用。你不需要现在就知道很多命令参数，也不需要理解完整协作工作流。

## 你甚至可以把操作委托给 AI

这一步完全可以继续沿用本书前面的思路：不是所有操作都要自己硬背，而是把目标说清楚，让 AI 陪你走。

比如你可以直接说：

```text
我现在准备开始给主页补内容。
在动手前，请先帮我做一个最小存档，让我之后改坏了还能退回来。
请一步一步带我完成，不要展开 Git 理论。
```

或者：

```text
请帮我查看这个项目目前有没有存档记录。
如果之后我改坏了，告诉我最小回退路径是什么。
只讲这次需要知道的部分。
```

这种说法很适合基础版。你不是在上 Git 课，而是在用 Git 解决一个马上会遇到的实际问题。

## 为什么有了存档，心态会明显不一样

一旦你有了这个最小闭环，后面的改动心态会完全不一样。你会更敢补内容，更敢试不同写法，也更敢让 AI 帮你做稍大一点的调整。不是因为风险消失了，而是因为你知道，即使某次改坏了，也不是世界末日，你有办法回到上一个稳定版本。

这一层安全感，往往比“我已经学会了多少 Git”更重要。
