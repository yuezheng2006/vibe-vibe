# F. Vibe Coding 的能与不能

基础版一直在强调一件事：不要神化 Vibe Coding，也不要低估它。它很适合很多场景，但不是所有项目都应该用同样方式推进。

## 它很适合这些事

| 场景 | 为什么适合 |
|------|-----------|
| 个人主页、作品集 | 反馈直观，容易快速迭代 |
| 原型验证 | 能很快把想法做出来，判断值不值得继续 |
| 内部小工具 | 范围可控，目标明确，适合快速落地 |
| 活动页、展示页、简单应用 | 页面和交互可以快速推进 |

## 它不适合被“轻率”对待的场景

| 场景 | 为什么要更谨慎 |
|------|----------------|
| 高并发核心系统 | 牵涉性能、稳定性和复杂架构 |
| 金融交易核心链路 | 风险高，错误成本极大 |
| 强合规、强审计系统 | 不是只做出来就够，还涉及流程与责任 |
| 高敏感数据系统 | 安全、权限、审计要求更高 |

这不是说 AI 不能参与这些系统，而是说这些场景不能只靠“先 vibe 一把”来推进。

## 一个很有用的判断标准

如果一个项目一旦出错，代价主要是“体验不好、要继续改”，那通常很适合 Vibe Coding 起步；如果一旦出错，代价会变成“资金损失、隐私风险、严重责任问题”，那就必须更谨慎，甚至需要更专业的工程与审查流程。

## 基础版真正想帮你建立的边界感

你现在已经有能力做出真实作品了，但不需要因此假装所有项目都该用同样方式做。知道什么时候该快速试、什么时候该慢下来、什么时候该求助专业支持，本身就是成熟的一部分。
