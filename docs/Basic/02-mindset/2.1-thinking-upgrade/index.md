# 2.1 为什么要把项目带回本地，以及如何把它导出下来

很多人做完第一个版本之后，会有一种很自然的想法：既然已经能预览、能点、能聊天，是不是先放着也可以？当然可以停在这里。但如果你想继续认真经营这个作品，就会很快碰到一个限制：它还不在你手里。

“不在你手里”不是说平台不能用，而是说你的掌控感还不完整。你也许能继续让平台帮你改，但你没有一个固定的位置放它，不知道它到底由哪些文件组成，也很难形成“这是我会继续维护的项目”的直觉。它更像一个暂存在别人工作台上的成果，而不是你自己的书桌上的项目。

把项目带回本地，真正改变的不是技术难度，而是关系。平台更像一个帮你快速搭出雏形的地方，本地工作台则像你自己的书桌。书桌不一定更高级，但它更稳定，也更可控。你知道东西放在哪里，知道下一次回来还能从哪里接着做，这种感觉会直接影响你后面能不能持续推进。

## 本地到底解决了什么

你可以先不用把“本地开发”想得太重。对基础版来说，本地至少解决四件很现实的事：

| 你会得到什么 | 为什么它重要 |
|------|----------------|
| 固定位置 | 你知道项目以后放在哪里，不会每次都重新找 |
| 长期可改 | 后面每一轮优化都有稳定落点，而不是停在平台聊天记录里 |
| 文件可见 | 你开始知道页面、样式、聊天相关代码大概分别在哪里 |
| 更可控 | 报错、改动、存档、部署都会更清晰地围绕同一个项目进行 |

所以这一节最实际的任务，不是理解什么复杂原理，而是先把项目放到一个正确的位置。

## 实际上怎么导出

不同平台的按钮名字会不一样，但路径一般都差不多：你会在项目页面附近看到 `导出代码`、`下载项目`、`同步代码` 之类的入口。你的目标不是研究平台实现细节，而是把当前这个能运行的版本完整拿下来。

建议按这个顺序做：

1. 先确认自己下载的是当前可用版本，而不是随手点到的旧草稿
2. 下载后，如果是压缩包，先正常解压
3. 给项目文件夹换一个你以后看得懂的名字
4. 把它放到一个固定目录，而不是长期丢在下载目录里

你完全可以简单一点，比如放在 `~/Projects/`、`~/Code/`、`桌面/我的项目/` 这样的固定位置。重点不在于目录名字多专业，而在于你以后还能一眼找到。

## 文件夹怎么命名更省事

别小看这个动作。很多人后面会因为文件夹命名太随意，自己都分不清哪个是正式版、哪个是测试版，改着改着就乱了。基础版不用搞复杂规则，但至少让名字有辨识度。

比较顺手的方式是：

```text
my-homepage
digital-twin-site
linan-homepage
my-site-v1
```

尽量避免这类名字：

```text
新建文件夹
项目最终版
项目最终版2
个人主页最新最终不改了
```

它们的问题不是“不专业”，而是几天后你自己也认不出来。

## 一个很值得现在就建立的习惯

从这一步开始，你可以把这个项目目录当成以后反复回来的地方。后面你加内容、改样式、调数字分身、准备上线，都会回到这里。所以这一步看起来像琐事，其实是在建立一种长期关系：这个文件夹不只是“刚刚下载下来的东西”，而是你以后工作的主场。

如果你现在心里还有一点犹豫，觉得“是不是先继续在平台里改一改也行”，我的建议是先别拖。你已经到了该把项目带回来的时候。因为第 2 章之后的重点，不再是“生成一个东西”，而是“持续把这个东西改好”。而持续修改，最好从自己的工作台开始。
